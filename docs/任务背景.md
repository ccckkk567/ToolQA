## 一、理解ReAct框架（论文核心）

ReAct = Reasoning + Acting 的交替生成。它的本质是扩展了LLM的动作空间：除了直接生成答案，模型还可以生成"思考"（Thought）和"工具调用"（Action）。

一次完整的推理链长这样：

1 Thought 1: 我需要先搜索X，找到Y，然后得出答案

2 Action 1: search[X]

3 Observation 1: (工具返回的内容)

4 Thought 2: 根据观察, X 是 …, 所以我接下来需要查 Z

5 Action 2: lookup[Z]

6 Observation 2: (工具返回的内容)

8 Thought 3: 综合以上, 答案是...
Action 3: finish[最终答案]

这与题目十一的输出要求完全吻合：问题分解 → 工具使用规划 → 调用了哪些工具 → 工具返回结果 → 最终答案。

## 二、对应题目十一的具体实现步骤。

#### 第1步：读懂输入：

自然语言问题：这就是ToolQA数据集里的问题

工具列表：相当于论文里的Action Space（search / lookup / finish）

你的Agent要能理解问题→规划用哪些工具→按顺序调用

### 第2步：构建Prompt（Few-shot）

这是ReAct方法的核心操作。论文用**少样本示例（few-shot exemplars）**告诉模型如何交替输出Thought和Action。

参考论文附录C的模板格式，你的Prompt应包含：

1 [示例问题]
2 Thought 1: 我需要搜索X, 找到Y
3 Action 1: search[X]
4 Observation 1: (示例观察结果)
5 Thought 2: X是..., 接下来需要...
6 Action 2: finish[答案]
7 ---
8 [真实问题]
9 Thought 1: (让模型继续生成)

关键设计原则（来自论文第3.2节）：

用不同类型的思考：分解目标、提取信息、常识推理、搜索改写、综合答案

HotpotQA用了6个示例，Fever用3个——示例数量不需要很多

示例要覆盖"搜索失败时如何改变策略"这种边缘情况

第3步：实现工具调用循环：

1 def react_agent(question, tools, max_steps=7):
2 context = build_prompt(question) # Few-shot + 当前问题

for step in range(max_steps):
    # 1. LLM生成Thought + Action
    thought, action, action_input = llm_generate(context)

    # 2. 解析Action类型
    if action == "finish":
        return action_input  # 最终答案

    # 3. 调用对应工具
    observation = tools[action](action_input)

    # 4. 将结果拼回context，继续下一轮
    context += f"\nThought: {thought}\nAction: {action}[{action_input}]\\nObservation: {observation}"

# 超过步数上限，回退到CoT-SC
return fallback_cot(question)

##### 注意几个工程细节：

论文设定HotpotQA最多7步，Fever最多5步，超过则回退

- 要处理工具返回空结果的情况（论文表2中23%的失败来自搜索无效）

防止模型重复生成相同的Thought和Action（论文提到的repetitive loop错误）

## 五、一句话总结：

题目十一的本质就是实现ReAct框架：让LLM在回答ToolQA问题时，不直接猜答案，而是通过Thought→Action→Observation的循环，主动调用工具获取信息，最终用 finish[答案] 输出结果。评分的三项指标（准确性、工具正确性、工具次数）分别对应：答对了没有、用对工具了没有、有没有浪费调用次数。
