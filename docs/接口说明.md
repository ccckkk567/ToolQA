## 2. 当前项目采用的 canonical tool set

当前项目以 **A 的 prompt 中实际出现的工具名** 作为 canonical 名称：

- `search`
- `sql_interpreter`
- `calculator`
- `finish`

其中：

- `finish` **不需要**由 C 提供实现，B 会在主循环里特殊处理
- C 需要提供的是其余“真实可执行工具”

如果 C 暂时使用的名称不同，例如：

- `lookup`
- `retriever`

则有两种处理方式：

1. **推荐**：C 直接适配成 canonical 名称  
2. **过渡方案**：通过 `ReactConfig.tool_aliases` 做别名映射

例如：

```python
config = ReactConfig(
    tool_aliases={
        "lookup": "sql_interpreter"
    }
)
```

注意：

- 模型最终输出什么工具名，主要由 A 的 prompt 决定
- 所以长期最好统一到同一套工具名，避免运行时出现 `unknown tool`

## 3. 给成员 C：工具侧接口说明

## 3.1 B 对 C 的最低接口要求

`react_agent()` 接收的工具对象形式如下：

```python
tools: Mapping[str, Callable[[str], Any]]
```

也就是说，B 需要拿到一个“工具注册表”：

```python
tools = {
    "search": search_fn,
    "sql_interpreter": sql_fn,
    "calculator": calculator_fn,
}
```

每个工具函数签名统一为：

```python
def tool_fn(tool_input: str) -> Any:
    ...
```

即：

- 输入：**一个字符串**
- 输出：任意 Python 对象都可以，但最好稳定、可序列化、可打印

---

## 3.2 C 需要如何调用 B

最小调用方式：

```python
from agent.react_agent import react_agent, ReactConfig

result = react_agent(
    question="2022年6月5日从MIA飞往LAS的AA2319航班的起飞时间是什么？",
    tools=tools,
    llm_generate=llm_generate,
    config=ReactConfig(),
)
```

如果 C 只是给 B 提供 mock 工具，那么 C 需要做的主要事情是：

1. 提供 `tools` 注册表
2. 保证工具名和 canonical 名称一致，或给出 alias
3. 保证工具输入是字符串
4. 约定好空结果 / 异常的行为

---

## 3.3 工具返回值如何被 B 处理

B 内部会把工具返回值转成 observation 字符串。

当前规则如下：

- `None` → `"No result returned."`
- `""` → `"Could not find results."`
- `str` → 原样返回（strip 后为空则视为空结果）
- `tuple` → `repr(tuple)`
- `list`
  - 空列表 → `"Could not find results."`
  - 若包含 tuple（例如 SQL 行结果）→ `repr(list)`
  - 否则 → `json.dumps(..., ensure_ascii=False, default=str)`
- `dict` → `json.dumps(..., ensure_ascii=False, default=str)`
- 工具抛异常 → `"ToolError: ..."`

### 建议 C 返回什么样的数据

最推荐：

1. `search`：返回 `str`
2. `calculator`：返回 `str` 或数值
3. `sql_interpreter`：返回类似：

```python
[("21:43",)]
```

因为这和 A 的 few-shot 示例风格一致。

---

## 3.4 B 如何统计工具调用

这一点 D 也会用到，C 也需要知道。

### 会计入 `tool_call_count` 的情况

- 成功调用了一个真实工具
- 工具报错但确实执行过调用

### 不计入 `tool_call_count` 的情况

- `finish[...]`
- 解析失败后的重试
- `loop_blocked` 的 synthetic observation
- 未知工具（因为没有真正调用到任何工具函数）

---

## 3.5 未知工具与 alias 的行为

如果模型输出的 action 不在 `tools` 中，且 alias 也解析不到，则 B 会回填：

```text
ToolError: unknown tool 'xxx'. Available tools: [...]
```

不会直接崩溃。

这意味着：

- C 如果尚未完成工具实现，也可以先给 mock
- 但工具名最好尽早对齐，否则模型会持续收到 unknown tool

## 4. 给成员 D：评测侧接口说明

## 4.1 `react_agent()` 返回什么

`react_agent()` 返回的是一个 `AgentResult` 对象。

如果 D 更喜欢 dict，可以：

```python
result.to_dict()
```

当前主要字段如下：

```python
{
    "question": str,
    "final_answer": str,
    "trace": list[StepRecord],
    "step_count": int,
    "tool_call_count": int,
    "used_fallback": bool,
    "finish_reason": str,
    "available_tools": list[str],
    "parser_error_count": int,
    "raw_final_output": str | None,
    "fallback_trace": list[FallbackSample],
    "parser_failures": list[ParserFailure],
    "fallback_low_confidence": bool,
}
```

---

## 4.2 给 D 的字段语义说明

### `final_answer`

- 最终答案字符串
- D 做 EM / F1 时，主比较字段应优先用这个

### `trace`

- ReAct 主循环的结构化轨迹
- 每个元素是一个 `StepRecord`

### `step_count`

定义：

- 统计**成功解析出的 ReAct step 数**
- **包含最终 `finish` 那一步**
- **不包含 parse retry**

### `tool_call_count`

定义：

- 只统计**真实发生的、非 `finish` 的工具调用**

不计入的情况：

- `finish`
- parse retry
- loop blocked
- unknown tool

### `used_fallback`

- 是否进入 fallback 阶段

### `finish_reason`

当前可能出现：

- `"finish"`：正常由 `finish[...]` 结束
- `"max_steps_fallback"`：超步数进入 fallback
- `"loop_fallback"`：循环检测触发 fallback
- `"parse_failure_fallback"`：解析连续失败触发 fallback

### `parser_error_count`

- 解析失败总次数
- 包含同一 step 内的多次重试失败

### `raw_final_output`

- 正常结束时：最后一次模型输出的原文
- fallback 结束时：被选中的 fallback 样本原文

### `fallback_low_confidence`

- fallback 多数投票不唯一时为 `True`
- 可供 D 在分析报告里标记“低置信 fallback case”

---

## 4.3 `trace` 的结构

`trace` 中每一项当前结构如下：

```python
{
    "step": int,
    "thought": str,
    "action": str,
    "action_input": str,
    "observation": str | None,
    "raw_llm_output": str,
    "status": str,
    "resolved_action": str | None,
}
```

### 字段说明

#### `action`
- 模型原始解析出的工具名

#### `resolved_action`
- alias 映射后真正命中的工具名
- 若没有 alias，则通常与 `action` 相同
- 若是 `finish`，则为 `"finish"`

#### `status`

当前常见值：

- `"ok"`：工具正常调用
- `"tool_error"`：工具报错或工具不存在
- `"loop_blocked"`：重复调用被拦截
- `"finish"`：正常结束

---

## 4.4 `parser_failures` 的结构

如果 D 需要分析 parser 问题，可以看：

```python
{
    "step": int,
    "attempt": int,
    "raw_llm_output": str,
    "error_type": str | None,
    "error_message": str | None,
}
```

适合用于统计：

- 哪类格式错误最多
- 模型是否频繁一次输出多组 step
- 哪些题更容易出现格式偏移

---

## 4.5 `fallback_trace` 的结构

如果进入 fallback，则会记录每次采样：

```python
{
    "sample_id": int,
    "raw_output": str,
    "final_answer": str,
    "normalized_answer": str,
}
```

适合用于：

- 查看 fallback 投票过程
- 分析平票 / 低置信情况
- 检查答案规范化效果

注意：

- `fallback_trace` **不混入** ReAct 主 `trace`
- D 如果做主流程统计，建议将两者分开看

---

## 4.6 D 侧建议如何计算

### 准确率相关

建议优先比较：

```python
result.final_answer
```

### 工具使用统计

建议直接使用：

```python

```

### 失败 case 分类

建议结合：

- `finish_reason`
- `trace[-1].status`（若 trace 非空）
- `parser_failures`
- `fallback_low_confidence`

做分类。

## 5. 当前已验证过的行为

B 已有单元测试覆盖以下行为：

- 正常主循环完成
- parse retry 不增加 `step_count`
- alias 映射生效
- unknown tool 不计入 `tool_call_count`
- loop blocked 后进入 fallback
- fallback 平票时按 first seen 处理
- fallback 会剥离 `finish[...]`
- 超长 observation 会被截断
